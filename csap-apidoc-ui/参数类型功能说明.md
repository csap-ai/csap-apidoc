# 参数类型功能使用说明

## 🎯 功能说明

现在 API 文档系统支持按照参数类型（`paramType`）区分显示和测试不同类型的请求参数了！

## 📋 支持的参数类型

| 类型 | 说明 | 颜色 | 示例 |
|-----|------|------|------|
| **PATH** | 路径参数 | 🔵 蓝色 | `/api/user/{id}` 中的 `id` |
| **QUERY** | 查询参数 | 🔷 青色 | `?page=1&size=10` |
| **BODY** | Body 参数 | 🟢 绿色 | POST 请求的 JSON 数据 |
| **FORM_DATA** | 表单数据 | 🟠 橙色 | 文件上传、表单提交 |
| **HEADER** | 请求头 | 🟣 紫色 | 自定义 HTTP Headers |
| **DEFAULT** | 默认 | ⚪ 灰色 | 等同于 QUERY |

## 🌟 主要功能

### 1. 参数分组显示

参数会按照类型自动分组显示，每个类型有独立的表格：

```
📦 请求参数

🔵 PATH    Path 参数
┌──────────────────┐
│ id  | int | 必填 │
└──────────────────┘

🔷 QUERY   Query 参数  
┌──────────────────┐
│ page | int | 可选│
└──────────────────┘

🟢 BODY    Body 参数
┌──────────────────┐
│ name | string   │
└──────────────────┘
```

### 2. 智能在线测试

在"在线测试"输入参数后，系统会自动：
- ✅ 替换 URL 中的 PATH 参数
- ✅ 将 QUERY 参数添加到查询字符串
- ✅ 将 BODY 参数作为 JSON 发送
- ✅ 将 HEADER 参数添加到请求头
- ✅ 将 FORM_DATA 作为表单数据发送

### 3. 导出优化

导出的文档（Markdown/OpenAPI/Postman）会按参数类型分组展示。

## 💡 使用示例

### 示例 1：用户详情查询（PATH 参数）

```
API: GET /api/user/{id}
```

**在线测试输入：**
```json
{
  "id": 123
}
```

**实际请求：**
```
GET /api/user/123
```

---

### 示例 2：文章列表（QUERY 参数）

```
API: GET /api/articles
```

**在线测试输入：**
```json
{
  "page": 1,
  "size": 10,
  "keyword": "技术"
}
```

**实际请求：**
```
GET /api/articles?page=1&size=10&keyword=技术
```

---

### 示例 3：创建文章（BODY 参数）

```
API: POST /api/articles
```

**在线测试输入：**
```json
{
  "title": "技术文章",
  "content": "文章内容...",
  "tags": ["技术", "教程"]
}
```

**实际请求：**
```
POST /api/articles
Content-Type: application/json

{
  "title": "技术文章",
  "content": "文章内容...",
  "tags": ["技术", "教程"]
}
```

---

### 示例 4：混合参数类型

```
API: POST /api/article/{articleId}/comment
```

**后端配置的参数：**
- `articleId` - paramType: **PATH**
- `page` - paramType: **QUERY**
- `content` - paramType: **BODY**
- `Authorization` - paramType: **HEADER**

**在线测试输入：**
```json
{
  "articleId": 456,
  "page": 1,
  "content": "这是一条评论",
  "Authorization": "Bearer token123"
}
```

**实际请求：**
```
POST /api/article/456/comment?page=1
Headers:
  Authorization: Bearer token123
  Content-Type: application/json

Body:
{
  "content": "这是一条评论"
}
```

---

### 示例 5：文件上传（FORM_DATA 参数）

```
API: POST /api/upload
```

**在线测试输入：**
```json
{
  "file": "...",
  "description": "文件说明"
}
```

**实际请求：**
```
POST /api/upload
Content-Type: multipart/form-data

FormData:
  file: [文件数据]
  description: "文件说明"
```

## 🔧 后端如何配置

在后端 API 开发时，为每个参数设置 `paramType` 字段：

```java
// 示例（伪代码）
@ApiParam(name = "id", paramType = "PATH", required = true)
@ApiParam(name = "page", paramType = "QUERY", required = false)
@ApiParam(name = "data", paramType = "BODY", required = true)
```

## 📊 参数表格显示

在 API 文档中，每个参数都会显示：

| 字段 | 说明 |
|------|------|
| 参数名称 | 参数的变量名 |
| 数据类型 | int、String、List 等 |
| **参数类型** | PATH、QUERY、BODY 等（带颜色标签） |
| 是否必须 | 是/否 |
| 默认值 | 参数的默认值 |
| 示例 | 参数示例值 |
| 备注 | 参数说明 |

## ⚠️ 注意事项

### 1. PATH 参数命名规则
URL 中的占位符必须与参数名一致：
```
✅ 正确: /api/user/{userId}  参数名：userId
❌ 错误: /api/user/{id}      参数名：userId
```

支持两种格式：
- `{paramName}` - 推荐
- `:paramName` - 也支持

### 2. 参数优先级
如果同时存在多种类型：
1. PATH 参数 → 替换 URL
2. QUERY 参数 → 添加到 URL 查询字符串
3. BODY/FORM_DATA 参数 → 作为请求体
4. HEADER 参数 → 添加到请求头

### 3. DEFAULT 类型
如果参数没有指定 `paramType`，会自动当作 **QUERY** 类型处理。

### 4. BODY vs FORM_DATA
- **BODY**: 适用于 JSON 数据（Content-Type: application/json）
- **FORM_DATA**: 适用于文件上传、表单提交（Content-Type: multipart/form-data）
